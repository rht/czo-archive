Tim Abbott (Mar 16 2022 at 20:14):

  • One of the biggest proposed changes is having the compose box width match the width of a rendered message. It is definitely annoying for the compose box to be wider than the display area, in that paragraphs that look like reasonable in your compose box can look way too high after sending; I definitely find myself adding paragraph breaks when editing messages for this reason. So I think this potentially does solve a problem.
  • I worry that it will create a couple new user interaction problems: (1) It'll make long URLs that one might paste into the compose box, including our Markdown syntax for uploaded files, more unwieldy feeling. (2) It's possible that this will make it hard to work with code blocks of LaTeX syntax that one might be working on in the compose box. Possibly this second problem could be solved by extending the existing "full-height compose box" feature to support increased width as well, and there's potential solutions to the first problem (including some that would be a very large investment, like handling uploads via a WYSIWIG mechanism).

One observation about the potential downsides of changing the width is that our "message edit" UI already has the same width as a message, so that may be a useful source of intuition for whether it'd in practice either of the worries mentioned in the second bullet are real issues.

Tim Abbott (Mar 16 2022 at 20:16):

Here's a couple ideas that would lean further into the direction of having this greater width.

  • I think it's possible putting your avatar into the compose box to to the left of the textarea would make the design for this less-wide compose box look cleaner. Obviously, it's redundant information to show your avatar there, but I think the idea would be that it increases the illusion of your drafting a message that will soon appear in the message feed.
  • We could also put "NOW" in the space to the right of the text box where a timestamp would be -- with some sort of UI indication that you can click this to change the send time as how we do "Send later". We'd need to play with whether there's enough space to clearly indicate the time a message would be sent when you've enabled "Send later", but this might be another opportunity to make the space we're spending here feel useful and thematic.

Tim Abbott (Mar 16 2022 at 20:19):

  • We'd need to have a mobile web design that's more different from the non-mobile web design, in that one wants all the width for the text box that one can get when composing a message on mobile. I think this is workable from an implementation complexity standpoint.

Tim Abbott (Mar 16 2022 at 20:30):

We have tried designs with that approach before, and the problem is that it makes one feel like sending a message asks you too many questions -- first what type of message is it, then what stream, then what topic. Switching message type is an infrequent action, and we probably want it to be that way (by having users usually start composing a message the other way), and it also just feels distracting for it to have so much visual prominence.

Tim Abbott (Jul 19 2022 at 21:35):

I'm not feeling great about the direction of having such a large STREAM|PM input element. There's a few concerns on my mind:

  • We currently have a visual pattern with the recipient bars in the message feed that you can just look at the left end of the recipient bar to read where a message was sent to, and having some extra text to the left breaks that pattern.
  • It's already visually obvious whether you're writing a PM or a stream message because of the 2-input vs. 1-input shape; I'd expect someone who's been using Zulip for more than a week to never find it useful to read the PM|STREAM labels.
  • The compose box design is otherwise very clean feeling, with essentially no text that stands out as something you're supposed to read other than the details of the message itself (recipient, content).

Tim Abbott (Jul 19 2022 at 21:36):

This is really tricky, because from a hierarchy standpoint, a STREAM/PM switcher should be to the left of the stream input.

Tim Abbott (Jul 19 2022 at 21:38):

I can think of a couple options that would address this issue:

  • Have the PM/stream switch element be hidden behind a 3-dot menu sort of thing in the corner. I think we probably don't want that, since an extra click is probably pretty significant in terms of the utility of this sort of widget.
  • Having it be represented by a single icon, with something indicating "switch", like https://fontawesome.com/v4/icon/random, and have the explanation of what it does be in tooltip (say, "Switch to private message" or "Switch to stream message").

  • Somehow the pipes/separators between the groups of icons feel like they're an icon for a compose option that happens to be a pipe. Maybe the separator icons should be in a slightly lighter font?

I think my final concern is I'm very reluctant to rework our closed-compose UI to be a small click target in the corner. Alya Abbott (Jul 21 2022 at 22:32):

We currently have a visual pattern with the recipient bars in the message feed that you can just look at the left end of the recipient bar to read where a message was sent to, and having some extra text to the left breaks that pattern.

Tim Abbott and I did a bit of brainstorming today on how we might address this comment, and here's a rough proposal we could chat about:

  1. As Vlad Korobov proposed in our chat earlier this week, the way to switch from composing to a stream to composing a PM could be a "Private message" option inside the stream dropdown (shown above all other options). So no dedicated button would be required.
  2. In that case, what happens when you're inside a PM compose view? If we continue to show "Private message" inside a dropdown, we still have a situation where there's extra text to the left of the addressee(s).

I don't know if it would feel good, but an idea we came up with is to replace the "Private message" text in the dropdown with a simple "To:". You could still click on the dropdown to switch to a stream name as your compose target. The field could be narrower in the PM compose situation and wider when composing to a stream.

  1. If we do something like (2), we wouldn't need a "To:" inside the addressee text box. For consistency, this might also cause us to remove "Topic:" from the topic name box.

  2. The result of all that is that we'd need to re-add some kind of dedicated "go to conversation" button.

Anyway, there's a lot of cascading consequences, but maybe worth discussing. A much more minimal change would be to just go with the "Private message" in the stream dropdown idea, without all the rest. :)
