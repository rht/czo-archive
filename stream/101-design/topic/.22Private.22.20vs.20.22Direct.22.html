Tim Abbott (Jan 06 2023 at 02:46):

For looking at what would be involved:

  • If we're OK with changing nothing about the API or data model (i.e. just change user-facing strings, and worry about changing internals later), there's only about 50 translated strings that contain "private message" in them in the Zulip server and web application that would need to change.
  • There's a bit of a transition messyness if we rebrand this and don't simultaneously do a self-hosted server release and mobile app release, in that it'll be hard to run a self-hosted Zulip installation with consistent terminology around this.
  • There's probably a lot of work on the Help Center etc. after a rename, but it's probably not hard.

Tim Abbott (Jan 21 2023 at 00:27):

Doing this sort of change if we've decided to do it has two significant benefits:

  • While having design conversations, we can use the new terminology. There may be places where something looks better/worse with the slightly different string lengths, for example.
  • When doing new feature work, we can use the new names (for example, today, I thought about a name for a file that is private_message_x that we'd probably want to call direct_message_x if we were migrating).
  • We can start figuring out how to do the API migration, which is the kind of thing that might take like a year of staged transitions to complete for backwards-compatibility reasons ("subject" still exists in a couple places in our API, and we renamed "subject" to "topic" in the UI like 8 years ago!). The first step or two of that sort of staged migration might be a small amount of work, followed by a lot of waiting.
