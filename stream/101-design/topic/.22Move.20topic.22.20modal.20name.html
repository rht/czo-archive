Evy (Oct 07 2022 at 17:27):

I think that renaming being the same as moving for a user is really confusing, and this is part of why I still think we should store topics with IDs and have renaming a topic not actually move any of the messages.

Tim Abbott (Oct 07 2022 at 22:07):

I understand you want to push that particular technical architecture change, but I'm not sure I see how it's relevant to the product question here. We don't intend to allow two __different__ topics with the same name; that would be incredibly confusing to say, display in the left sidebar.

Just about any feature one can build with a "topic IDs" data model can also be built with the existing "topics are strings" data model. A possible exception is generating a link to a topic that (1) isn't itself an object in the database and (2) still works if the topic is renamed. (Even with topic IDs, it's still tricky to handle corner cases like splitting a topic into two topics, so they're not a panacea).

Tim Abbott (Oct 07 2022 at 22:07):

From a user/product perspective, "renaming a topic" and "moving a topic within a stream" are just different ways of expressing the same desire -- to have the topic's name be different from what it was before. The reason we prefer the term "move topic" is that if you want to merge a topic into another topic, for example, that's a better term than "rename".

Tim Abbott (Oct 07 2022 at 22:11):

I think "Move or rename topic" is a reasonable name if we want to hint both use cases; but I think having separate menu items for "move" vs "rename" seems like it'd just add clutter.
