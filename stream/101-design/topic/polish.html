Topic: polish

Wesley Aptekar-Cassels (Jun 26 2021 at 01:41):

I started a conversation in #feedback > emoji symbol for streams about UI and UX polish, and how I think that's something Zulip has underinvested in — I think it's worth starting a thread to chat more about that and how we want to think about UI/UX polish work.

In general, I'm coming at this from a place of believing that Zulip's threading model, commitment to a good experience for power users (keyboard shortcuts, etc), and unique features (LaTeX, linkifiers, etc) make it a much better tool for team collaboration than anything else out there. However, the most common feedback that I hear about Zulip is that it's confusing and difficult to use. It's convenient to blame that on the threading model, but I'm not convinced by that: there are a bunch of things that are idiosyncratic about Zulip's design in 2021 (some for better, some for worse), and most of the things that are idiosyncratic are not fundamental to the design of Zulip.

Zulip has way, way more features than most of our competitors:

  • Timezone-aware dates (like 2021-06-25T06:00:00-07:00)
  • LaTeX
  • Spoilers
  • Polls
  • Todo lists
  • Linkifiers
  • Alert words
  • Marking topics as resolved
  • Silent mentions
  • Surely, many more things I'm forgetting

These are all really great features! However, most of these things could use some love to take them from things that are hard to discover and annoying to use to things that are a joy to use and differentiate Zulip from other team chat apps.

Let's look at a couple examples. I'll note that while I'm going to be complaining about things here, I really appreciate all the work that has gone into these features — I'm starting this conversation because these are great features, that just need a bit more love to make them more usable and intuitive.

Polls

Polls are a super useful feature — they're something we had a bunch of requests for, and for good reason. We implemented them, but...

  • They still aren't implemented in mobile, years after we released them on web/desktop
  • They don't scale up to huge numbers of users, since we display all of the usernames.
    • We don't even display the usernames in a very useful way — they're all in a line of text, so there's no easy way to scan the list and see if a specific person responded.
  • We don't offer any options, despite plenty of requests for different types of polls.
    • No way to limit the set of responses
    • No way to have anonymous polls
    • No way to set an end date for a poll
  • The CSS looks pretty outdated, even relative to the rest of the design of Zulip.
  • We don't have a easy way to visually see which option is winning. Check out a twitter poll — that gives bars as the background for each option after you vote, so you can see how the options stack up without reading every number.
  • There's no easy visual way to make a poll — you have to use the /poll command, which the "Preview" option doesn't work on, so there's no way to tell that you've gotten it right except sending it.

This stuff isn't a lot of work to implement! We could spend a week of work just polishing polls, and implement most of these. But we haven't, several years after we introduced polls. That's sad — I'd love to use polls more, but they have problems that make me often prefer to just do an emoji-reaction-vote.

Timezone-aware date mentions

Timezone-aware dates are a super, super useful Zulip feature. When was living in Taiwan and interacting a lot with NYC people on Zulip, I really appreciated being able to talk about times without manually converting timezones and mentioning two or three times in every scheduling message. However, there are a few pain points:

  • They're super hard to discover. I saw them several times, and had to "view source" on a message someone else sent to figure out how to use them. We should have a clock icon next to the smiley face button on the compose box to insert a timezone-aware time.
  • The time picker is annoying to interact with
    • It doesn't respect your 12/24 hour setting
    • If you put in something like "15" for the hour position, it shows up as red, but when you press tab, it converts it to "03" (correctly), but doesn't convert the "AM" to "PM" (incorrectly)
    • You can't toggle the "AM"/"PM" selector with the keyboard
  • The time picker CSS looks sort of janky — things are aligned strangely, and the arrows for increasing/decreasing numbers overlap strangely.
  • On the tooltip, we show the original text, but that isn't actually very useful — we should show it in the timezone that the author specified it in, but parsed and shown in a friendlier way than the format with "T", "Z", UTC offsets, etc.

Again, none of this stuff is super hard to fix! Someone could spend a week on this and make it a really nice experience! But we haven't done this.

It's worth thinking about this as an investment — part of the reason we have so many features is that we don't spend a ton of time on each feature. But given that the main negative feedback we get about Zulip is that it's difficult to figure out how to use, I think it's worth spending more time polishing the features that we do make. I think spending a handful of days or a week looking at an existing feature and figuring out how to polish it and make it nicer and more intuitive to use is a really valuable use of time, and I'd encourage folks who are interested to do that :)

I'm also curious to hear other folks thoughts here ­— is this something people agree with?

Tim Abbott (Jun 29 2021 at 17:43):

Thanks for all the thoughts, folks! The beginning of this thread was really interesting, but it seems to have digressed a bit and had an unfortunate shift in tone, so let me respond to the general discussion.

  • I absolutely agree that polish is really important. If I had to guess, a majority of everyone's time goes into things that can be classified as "polish", in terms of how users experience it. One thing that is not obvious is just how much work making a polished feeling modern application is. I think we've all used plenty of UIs from extremely well funded companies that are super buggy, look weird, etc. And most of those companies have like 5x less UI surface are that Zulip does.
  • Given that it's possible to spend multiple person-months polishing a single feature (especially if one considers "implementing all feature requests for polls" as polish), it's inevitable that we have a lot of features that are not "fully polished".
  • We're actively working to hire more design/frontend leadership, which should increase our capacity for doing some classes of this work.

Tim Abbott (Jun 29 2021 at 17:54):

As one broader general point, I think talking about alpha/beta/etc. as statuses for features (or complaining that a feature hasn't been fully polished X months after first implementation) isn't a good way to think about the status of a feature. For one thing, there are competing priorities, and sometimes it takes some after a feature is first merged for us to know whether it's useful and something we want to build on.

We as a project have literally hundreds of features we can work on, and work tends to happens in bursts of effort where someone spends a few weeks on a specific feature, generally also involving a lot of conversations in this stream and elsewhere to figure out a path forward that is actually better. (I think all of us have seen attempts at polishing the product that we can't merge because they end up not working out how we'd imagined beforehand).

A better way to think about a feature is using more descriptive terminology:

  • Nonexistent: Zulip does not have the feature.
  • Checkbox (it's there, you can find it by reading our documentation and/or seeing other folks using it, and it might not be nice, but you can do it if you're determined to do so). Checkbox features are really important competitively, because "Zulip doesn't have polls" is a much more significant objection than "Zulip's polls aren't as nice as I'd like".
  • Complete: It can do all the things you naturally might want to do with that sort of widget (E.g. anonymous polls, editing, and other fancy things).
  • Reasonably bug-free: To what extent are there visible bugs one might encounter using the feature.
  • Polished: It looks nice and feels pleasant to use.
  • Discoverable: It's easy to find the feature contextually within the app.

Tim Abbott (Jun 29 2021 at 18:04):

So I think part of why folks were talking past each other about polls above is that currently polls are a checkbox feature that is moderately bug-free, not polished, not complete, and only socially discoverable (once you see one, you can figure out how to do it by googling if nothing else). That's still super valuable, and I'm very glad we have them even if their current state!

I think it's reasonable to discuss (on a dedicated thread) to what extent we want to change one or all of those aspects of polls in particular. But I think precise terminology is very helpful for having a discussion about the status of a feature. (I believe there is active work on the compose box that would support having a "poll button").

Wesley Aptekar-Cassels (Jun 29 2021 at 20:41):

As much as I agree that "general look and feel" is an area that we get criticized on and can certainly improve on, it's kind of overly, um, general. Whereas the next two are something where one individual can really do a deep dive.

For instance, here's something we can have a general discussion about in a productive way, I think. We've identified a problem — people are unhappy with the "general look and feel" of Zulip. What's our strategy for fixing that?

I think this is an interesting thing to look at, since there are several parts to it, and I think we may be blocking some of the easy parts on the hard parts unnecessarily. I see the "hard parts" here as coming up with a cohesive and attractive visual style for Zulip, building a library of UI components that fit that theme, and switching the whole app to using them. That's a lot of work, which will require a designer, lots of discussion, lots of sort of tricky frontend work, etc. But there's also a lot of more tactical work that we can go in order to move in a better direction, even if we aren't sure what the exact destination is.

And, to be fair, a lot of this is work that we're doing! A lot of the chatter in #design is about this sort of thing. But I think we could stand to do more of trying to look at the UI with fresh eyes, and looking for things that could be improved. I think one thing we could do here is try to make reporting small papercuts lower friction — perhaps a running topic in #feedback for people to post things that feel too small to start a new thread for. For instance, #19053 is something I noticed just now which feels a little silly to make an issue for since it's so small, but is absolutely something we should fix, since it looks pretty embarrassing as soon as you notice it. #19055 is similar — it's a small thing which is easy miss or to not follow up on, but if we do around a dozen things of that size, I think the app will feel noticeably nicer and less ugly.

Even as someone who is employed by Zulip, I often run into things that look a little funky, but it just feels like a lot of effort to make an issue, so I put it off and then forget.

We could also try things like having a "frontend polish week", where people dedicate time to looking for things to polish and then implementing them.

Wesley Aptekar-Cassels (Jun 29 2021 at 20:48):

I would also be happy to take a shot at writing up some general UI principles, if that's something we think would be useful. I do think there's a lot of stuff that we get right (being keyboard friendly, responsive, avoiding things popping up without user interaction, etc), but there are also things we can improve on.

I suspect that a broader plan for coming up with a more cohesive UI language for Zulip would look something like:

  • Come up with design principles (things that we care about that our UI should enable and strive for)
  • Come up with a frontend style guide, including things like font sizes, icon sizes, and a library of UI widgets for different usecases
  • Choose a small part of the UI to start converting to the style we're aiming for, so that we can talk about it in a concrete context
  • Iteratively discuss the style guide and implement it in more places, making sure that when we make decisions that are generic, we write them down.

Obviously this is something that it would help to engage a designer in, but I think there's value in starting that now, and I'd be happy to start a doc for it if it's something people think is valuable.
