Topic: hide topic editing pencil in interleaved views?

Alya Abbott (Sep 05 2023 at 22:32):

In a customer support discussion, it came up that the action of the "Edit topic" pencil button in the topic header bar is quite confusing in interleaved views. It's not very clear that editing the topic there will move all the messages in a topic, which may be above or below other messages that are not being moved.

Should we perhaps hide that button in interleaved views? Users will still be able to move messages via the message and topic menus as usual.

(I'm not caught up on #design , so apologies if there's discussion elsewhere that makes this question moot.)

Vlad Korobov (Sep 05 2023 at 23:12):

I vote for hiding it unless hovered or hiding those buttons unless inside a topic view

Vlad Korobov (Sep 05 2023 at 23:17):

The proposal we’ve discussed in the past is to show buttons in recipient bar only when in that topic view + possible to show those buttons in other view on hover. That was UI clean up. But if we think that those buttons could be misguiding without the full context of the topic we could have only one button in interleaved view “go to the full version of that topic”.
I would still appreciate mute control inside interleaved view. And I’m not sure it is ok to not show some options over an topic if a user doesn’t see all of it
