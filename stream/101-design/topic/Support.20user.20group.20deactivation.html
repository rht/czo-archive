Zixuan James Li (Jul 21 2023 at 06:16):

Proposal:

When a user group is deactivated, we set the field "deactivated" to True.
There will be some side effects as a part of the deactivation. We should also
modify changes to other systems that user groups interact with. Details
discussed below, where the related fields are parenthesized:

Group-group memberships (direct_subgroups/direct_supergroups)

When deactivated, the deactivated group should be removed from existing
group-group memberships. Or, to keep the memberships, we just skip deactivated
user groups in recursive queries. This will disable the DAG constraint check,
so we must re-check the constraints when reactivating the user group. The
downside is that it is harder to give an actionable error message when
reactivation fails due to discovered cycles.

Group based settings (can_mention_group/can_remove_subscribers_group)

When deactivated, can_mention_group, can_remove_subscribers_group, etc. need to
be reassigned to another group before deactivation. In the UI, we can
potentially prompt the user to set them back to the defaults either through a
separate request or a flag to the deactivation endpoint.

Audit logs (modified_user_group)

Audit logs should be unaffected. We record USER_GROUP_DEACTIVATION and
USER_GROUP_REACTIVATION events.

Mentions (mentioned_user_group)

Modify MentionData so that deactivated user groups are not included.
We can do something similar to extracting get_linkable_streams and have
get_linkable_user_groups.

I'm not too sure about the handling of missed messages for this one. Should we
skip notifications for deactivated user groups? When should the check happen?

Questions

  • Should we allow deactivating system groups?
  • Should changes to other fields, including name, description, and direct members be allowed?
  • How should deactivation interact with user group mentions in messages?
  • What are some other use cases of UserGroup that might be affected?
