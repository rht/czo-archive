Steve Howell (Apr 03 2019 at 20:09):

I think the only way to do this in a fairly straightforward way is to check for "@" at the start of the search string, and that would be our clue that you're searching for domains. (I'm wary of always looking for domain matches, because that can be super noisy.)

Steve Howell (Apr 03 2019 at 20:21):

I agree. Feel free to open an issue. @Rishi Gupta do you see any pitfalls in having searches like "@gmail" trying to match domain names? (It should be fine performance-wise, and I can't think of any usability issues. I don't mind if requiring "@" a bit inconvenient or hard to discover; my bigger concern is breaking behavior for normal searches.)

Rishi Gupta (Apr 03 2019 at 20:25):

The current user typeahead match function is a bit crazy, which makes it a bit hard to think about (product wise; I expect this isn't a hard change technically).
Ideally I'd like to move the stream and user typeahead to just be substring based, which will make domain matching more noisy.

I could eventually imagine adding search keywords like email:foo or profile:foo (for searching custom profile fields) though.
