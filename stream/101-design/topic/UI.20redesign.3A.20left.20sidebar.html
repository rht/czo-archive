Tim Abbott (Sep 21 2022 at 19:37):

I see the appeal of having the whole region be scrollable; it feels simpler. But I worry about some of the practical implications:

  • I'm not sure it's worth a line of space for a MORE/LESS control to hide a few of the global filters, so I'd prefer we focus on other directions for that detail.
  • I'm not sure that the "^ UNREAD" indicator addresses the user story for why we want the "All messages" and "Private messages" unread counts to be always visible, which is the situational awareness, without having to click on anything, of whether you have new unread messages, and whether new ones are coming in regularly. (The "All messages" total will be more important in lower-traffic communities). I've seen other products, like Discord, that do a "More unread above/below" thing in their sidebar, and if you've got a moderate number of unreads, it kinda feels useless, because it's always there. Possibly there's a tweak on it that could address it -- like if "12 unread private messages" fit? But that may be unfun to implement around corner cases like scroll positions where part of the "private messages" section is visible.
  • I feel like "Drafts" and "private messages being accessible with a single click is fairly important to those features feeling usable; having to scroll up potentially past 50 things to get to them likely won't feel good, and I'm not sure users will intuit that clicking "^ UNREAD" is an intended way to get to drafts/recent conversations/etc.
  • We should keep in mind we're likely to add an Inbox view adjacent to Recent Topics in the sidebar, and it's probably similarly going to be something that some folks want to visit often. (I expect it to be an option for the default view, and so some users will get there via clicking the Zulip logo in the corner, but others might still have Recent Topics as their default view, for example).

Tim Abbott (Apr 07 2023 at 18:21):

Yeah, stream groups is a feature that we've wanted to have for a while, but would be difficult to fit into the current left sidebar design. To talk a bit about the concrete changes in this proposal:

  • The top navigation views area will become collapsable by the user, with a custom collapsed state that has icons on it so you can still access things without expanding it in one click if you like; this is intended to balance the fact that these are very useful views but also take up a decent amount of space.
  • The current STREAMS section will become collapsible; this is mostly an irrelevant feature until we add stream group support, but in theory, stream groups would be a straightforward setting to add. There'd be an "OTHER STREAMS" section at the bottom if you have at least one stream group and not every stream is in a stream group. (I think stream groups would not be hard to start adding -- a base version would be just a data model and a dropdown in "Manage streams" for which group something is in, and we very much want them for permissions model reasons eventually).
  • Pinned streams become pinned only within their stream group. This is probably less convenient if you want to have a single place for all the streams you care about across the organization, but is likely more convenient in other settings. (Not an actual change until organizations actually have stream groups).
  • We're basically going to lose the global "filter streams" widget for organization using stream groups; instead you'll be only able to filter streams within a group, mainly because it's really not clear where to put a filter box otherwise. I think this is OK.
  • The main always-visible place for unread counts will possibly be on the stream group headings, not "All messages" or "Inbox", since the latter may be collapsed. But we can of course fiddle with this sort of detail.

Karl Stolley (Sep 21 2023 at 15:02):

My motivation for asking is that I think it might be smarter to move from the current flex-based layouts to a grid-based layouts on the left sidebar rows. There are a few advantages to that that immediately come to mind:

  1. Items, like checkboxes or icons/indicators, can be added and removed at the DOM level without affecting the layout
  2. Especially for things like multi-line topic names and DM groups, we might have better control over the vertical alignment of different elements
  3. The columns of each row can be more precisely sized and controlled at the container level (using things like minmax()) without having to dive into individual children and make different declarations for the flex grow, shrink, and basis properties.

Karl Stolley (Sep 21 2023 at 15:03):

I should add that my thinking about this is sort of top-level for each row. Certain elements, like the username/status emoji for one-to-one DMs, would be described as flexboxes. But anywhere we could basically draw a vertical line down the design seems to me to be a candidate for grid.
