Tim Abbott (Feb 23 2022 at 20:31):

Before we get too distracted in debating questions that don't affect our product direction here, here's my thinking on the core context here:

  • We want to add support for pronouns; I think everyone agrees with that.
  • The core design question here is whether the features we're discussing for pronouns are things that one might want for another custom profile field. If so, it's likely less technical work to extend the custom profile fields feature to support the generally useful features we want a Pronouns field to have, and then have potentially some sugar on top of that to make it easy to create such a field. (In part because custom profile fields provide a lot of useful infrastructure already).

Steve Howell (Feb 23 2022 at 20:33):

I think it is easier to solve concrete issues than abstract issues. Our profiles are already overly abstract.

Tim Abbott (Feb 23 2022 at 20:37):

Specifically, in terms of generic features I would start with:

  • We can add a generic custom profile fields feature to show up to 2 custom profile fields in the small user popovers. (Already filed as #21215)
  • We can add a generic custom profile fields feature to allow short text fields to have some easily chosen options; basically that's a hybrid of the existing Select and Short Text field types. I believe the details of this are a good discussion topic.

Those are both self-contained changes that would provide infrastructure needed for this feature. And from there, we can readily implement whatever UX we want for custom profile fields. (Including the option of having new organizations start with a Pronouns custom profile field or whatever other direction we decide we like). But my understanding of the technical side of this is that we'll want to build on top of custom profile fields for data storage and synchronization for any version of a Pronouns feature.

Tim Abbott (Feb 24 2022 at 19:57):

A few reasons:

  • The small popover has a bunch of options for common interactions you might want to do with another user, like reply to a message mentioning them.
  • It's often helpful to be able to still see messages at the same time as that content / those interactions.
  • The full profile information has a bunch of lower value information that one wants to be able to access, but isn't important to access often.
  • Fullscreen modals (effectively our "overlays) that cover most/all content are a design pattern one wants to avoid for lightweight operations (like "I want to know a bit about this user and maybe mention them").
  • Our current "full profile" modal is much uglier than the "user information" popover :smile:.
